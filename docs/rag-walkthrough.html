Real end-to-end walkthrough (example topic)

Example topic: Lithium iron phosphate batteries for grid storage (2024-2025)

  1. Frontend opens stream - UI creates an EventSource to GET /api/research/stream and resets state.
  2. Backend accepts request - FastAPI validates topic and starts the SSE pipeline.
  3. Query expansion - Fast LLM generates 3 diversified search queries.
  4. Search + URL harvesting - Tavily search returns snippets and top URLs; URLs are deduped.
  5. Concurrent scraping - Up to 4 URLs are fetched and cleaned; each source is capped.
  6. Report generation - Writer chain streams report chunks to the UI.
  7. Critique - Critic chain returns actionable feedback.
  8. Done - SSE emits done; UI saves history.

SSE event stream deep dive

Event schema

{
  "step":   "search | scrape | report | report_chunk | feedback | cache | done | error",
  "status": "processing | complete | streaming | error",
  "data":   "string payload (may be empty)"
}

Typical cache-miss sequence (illustrative)

  • search:processing
  • scrape:processing
  • search:complete
  • scrape:complete
  • report:processing
  • report_chunk:streaming (many times)
  • report:complete
  • feedback:processing
  • feedback:complete
  • done:complete ({"cached":false,"duration_ms":12345})

SSE example payloads

data: {"step":"search","status":"processing","data":""}

data: {"step":"search","status":"complete","data":"..."}

data: {"step":"report_chunk","status":"streaming","data":"# Title\n"}

data: {"step":"done","status":"complete","data":"{\"cached\":false,\"duration_ms\":12345}"}

Query expansion + scraping deep dive

  • Query expansion - fast LLM outputs 3 distinct query lines.
  • Search - Tavily provides snippets and top URLs per query.
  • Dedup - URLs are deduplicated while preserving order.
  • Scrape - up to 4 URLs are fetched concurrently via httpx.
  • Extract - content is taken from article or main when available.
  • Cap - each source is capped to reduce tokens.

Core pipeline (Python) explained

This section focuses on backend execution and the main.py pipeline flow.

main.py: app setup and shared state

  • Environment and logging: loads .env and configures structured logs for tracing.
  • Shared HTTP client: creates one httpx.AsyncClient in lifespan and closes it on shutdown.
  • Cache: _result_cache is guarded by _cache_lock with _CACHE_TTL and _CACHE_MAX.
  • FastAPI app: defines title, version, and registers GZip and CORS middleware.
  • Static serving: mounts frontend assets from frontend/dist when present.
  • Request ID middleware: sets X-Request-ID for every response.
  • Schemas: ResearchRequest validates topic length and ResearchResponse standardizes output.
  • Operational endpoints: /api/health, /api/cache/clear, and /api/config.

main.py: SSE pipeline (_sse_pipeline)

  1. Cache check: if a cached entry exists, emit cache and complete events, then return.
  2. Emit search and scrape processing events and run pipeline._stage_search_and_scrape.
  3. Emit report processing and stream writer chunks as report_chunk events.
  4. Emit report complete, run pipeline._stage_critique, and emit feedback complete.
  5. Cache the final payload and emit done with cached flag and duration_ms.
  6. On error, log the exception and emit an error event.

main.py: JSON endpoint (/api/research)

  1. Validate the request body with ResearchRequest.
  2. Check cache and return a cached ResearchResponse when available.
  3. Run pipeline.run_research_pipeline_async on a cache miss.
  4. Store the result in cache and return the JSON response with timing.

pipeline.py: staged execution

  • _in_thread: runs sync chain or tool calls in a thread to keep the event loop free.
  • _stage_search_and_scrape: expands queries, searches Tavily, collects URLs, dedupes, and scrapes top URLs concurrently.
  • _stream_write: yields writer-chain chunks from astream for streaming output.
  • _stage_write: collects chunks into the final report with a safe fallback on errors.
  • _stage_critique: runs the critic chain and strips conversational preambles.
  • run_research_pipeline_async: orchestrates stages and returns a dict for caching and responses.

tools.py and agent_modules: supporting modules

  • tools.py: Tavily client, async scrape helpers, HTML extraction, and search utilities.
  • agent_modules.llm: returns fast and writer LLM clients for different cost and quality tiers.
  • agent_modules.query/writer/critic: build chains used by the pipeline stages.

System design

High-level architecture and tradeoffs for the multi-agent research pipeline.

APIs and contracts

  • Streaming API: GET /api/research/stream returns text/event-stream with step, status, and data.
  • Batch API: POST /api/research returns ResearchResponse with search_results, report, and feedback.
  • Health API: GET /api/health reports LLM tier status, Tavily key, internet, and cache stats.
  • Config API: GET /api/config exposes public Supabase URL and anon key.
  • Error contract: HTTP 400 for invalid topics and HTTP 500 for pipeline failures.

Data flow

  1. Client opens EventSource or posts JSON with a topic.
  2. API validates input and checks the in-memory cache.
  3. Search and scrape run concurrently and emit progress events.
  4. Writer streams report chunks while the UI updates in real time.
  5. Critic generates feedback after the report completes.
  6. Result is cached and the done event or JSON response is returned.

Scaling and tradeoffs

  • SSE vs batch: SSE improves perceived latency but keeps connections open longer.
  • In-memory cache vs shared cache: local is fast but per-worker; Redis enables cross-worker reuse.
  • Concurrency vs rate limits: more scraping parallelism reduces latency but risks 403s and throttling.
  • LLM tiering: fast models cut cost and time; larger models improve report quality.
  • Sticky sessions for SSE: helps keep streams on one worker but reduces load balancing flexibility.
  • Input caps: MAX_RESEARCH_CHARS and MAX_CRITIC_CHARS control cost and response time.

schema.sql walkthrough

  • Create table search_history with columns for topic, results, report, feedback, and created_at.
  • user_id references auth.users(id) with ON DELETE CASCADE for cleanup.
  • Enable Row Level Security (RLS) to restrict access by user.
  • Policy SELECT: users can read only their own rows.
  • Policy INSERT: users can insert only their own rows.
  • Policy DELETE: users can delete only their own rows.
  • Indexes on user_id and created_at speed up history queries.
  • Realtime publication enables live updates if needed.
